# 全规则自动校验

## 适用范围

统一校验覆盖仓库发布的全部九张规则表：Google、GoogleCN、GoogleAI、
AI、YouTube、TikTok、BiliBili、Game 和 GameCN。每次生成都会在对应
JSON 报告中写入结构一致的 `verification` 区块，仓库总校验会检查这些
区块是否齐全、计数是否一致，以及人工审核指纹是否合法。

## 自动判断分两层

第一层检查正式生成过程。v2fly 仍是唯一自动生成上游；程序自动记录已
发布、按产品边界排除、按安全策略拒绝、由规则顺序消解，以及无法安全
转换的候选。只有最后一类进入人工审核。

第二层检查只读参考源。Google、GoogleAI、AI、YouTube、TikTok 和
BiliBili 会同时利用可用的 Sukka、BlackMatrix7 参考数据：

1. 已被正式产物覆盖：自动确认。
2. 已有最小范围、证据链接和处理结论：按
   `references/verification/reference-decisions.json` 自动复核。
3. 仅一个参考源出现：记录为单一参考差异，自动结束，不改变产物。
4. 至少两个独立参考源共同指出、但产物未覆盖：进入人工审核。
5. 已确认的精确主机没有实际出现在目标产物：进入人工审核。

“自动结束”不表示把参考源条目自动加入规则，而是保持正式上游边界，
避免单个第三方列表扩大分流范围。参考源的完整规则不会持久化；报告只
保留已确认例外和仍需人工决定的最小集合。

## 人工审核如何工作

人工不需要逐个重新搜索全部差异，只处理报告中的 `manual_review`：

- `manual_review_count`：本次真正未决的数量；
- `manual_review_fingerprint`：未决集合的稳定 SHA-256 指纹；
- `manual_review`：域名、共同指出它的参考源和触发原因。

审核后若决定加入，只添加最小 `DOMAIN` 或 `DOMAIN-SUFFIX` 补丁；若
决定排除或只接受精确主机，则把结论、作用范围和 HTTPS 证据写入统一
决定目录。以后程序会自动重放并检查这个决定，无需重复人工查询。

DNS、HTTP 可达性和出口观测会随地区、CDN 与时间变化，只能作为辅助
证据，不能单独决定 DIRECT、代理或服务归属。

## 自动合并边界

下列情况不能视为低风险：

- 人工审核集合相对上一版增加、替换或发生其他变化；
- 正式规则被删除或变化率超过阈值；
- 不支持语法集合变化；
- 决定目录无效，或精确主机决定没有落实到产物；
- 任一规则集缺少统一校验报告。

即使所有门禁通过，也只有仓库变量 `AUTO_MERGE_LOW_RISK=true` 时才允许
工作流自动合并。观察期应保持为 `false`。

## 审核通知

七条同步工作流只在分类为 `review-required` 时通知仓库所有者：

1. 自动 PR 写入一条风险摘要、审核原因和审核指南。
2. 首次出现该风险时，请求仓库所有者审核 PR，由 GitHub 产生
   `review_requested` 通知。
3. 同一 PR 已经等待审核、但风险内容发生变化时，通过一次 `@` 提及重新
   通知。
4. 通知指纹由完整风险评估和未决/不支持条目计算；相同指纹不会重复提醒。
5. `low-risk` 和没有产物变化的运行不发送审核通知。

邮件由 GitHub 的个人通知设置控制，仓库不保存邮箱地址、SMTP 密码或邮件
服务令牌。若要接收邮件，需要在 GitHub Notifications 中为
`Participating` 启用 `Email`。
